# 实验室样本交接看板（Labs Handover Dashboard）

本地实验室样本交接追溯系统，支持样本登记、批量交接、退回重交、签收、温控异常告警、角色权限、审计历史和筛选导出。

## ✨ 核心功能

| 功能模块 | 说明 |
|---|---|
| 🔐 角色权限 | **采样员（SAMPLER）**登记/发起交接，**接收员（RECEIVER）**签收/退回，**管理员（ADMIN）**仅可作废、审计、全量查看 |
| 📦 样本登记 | 批量登记，自动创建批次，**初始温度必填**，缺失则拒绝 |
| 🚀 发起交接 | 按批次指定接收人发起交接；交接前校验温控完整性 |
| ✅ 签收 | 只有**指定接收人**可签收；**严禁替签**；签收时每个样本需提供温度记录；**已签收后不可退回、不可改交接信息** |
| ↩️ 退回重交 | 仅**IN_TRANSIT（交接中）**状态可退回；退回**必须填写原因**，缺原因直接拒绝；**已签收样本不可退回**；退回后样本返还给采样员，可重新发起交接 |
| 🌡️ 温控异常 | 每次记录温度自动检测阈值（过高/过低），生成异常告警；支持标记处理 |
| 🔍 完整追溯 | 每个样本可查**当前保管人**、**状态**、**全部温控记录**、**完整交接链时间线**（登记→交接→签收→退回→重交→作废） |
| 🛡️ 审计日志 | 记录所有操作的操作者、时间、目标、前后状态、失败原因；**失败操作也记录** |
| 📊 筛选导出 | 按批次状态/样本状态/关键字/温控异常过滤；CSV 导出内容**严格匹配筛选条件**，包含批次/样本/温控/交接链四张表 |
| 💾 数据持久化 | JSON 文件存储（`data/db.json`），重启服务历史数据与退回再交接时间线**完全保留** |
| ⚠️ 原子操作 | 所有操作先校验再执行，**失败不写入半截记录** |

## 🚀 快速开始

```bash
# 1. 安装依赖
npm install

# 2. 启动（前端 + 后端同时启动）
npm run dev

# 或分别启动
npm run server:dev    # 后端：http://localhost:3001
npm run client:dev    # 前端：http://localhost:5173
```

默认打开 `http://localhost:5173` 登录即可。

## 🧪 功能验证步骤（完整测试清单）

### 1️⃣ 场景一：正常流程：登记 → 发起交接 → 签收

1. 登录 **sampler1 / sampler123**（采样员）
2. 左侧菜单 **样本登记**，添加 2 条样本：
   - 样本编号 `S-TEST-001` / `S-TEST-002`，初始温度 4℃（正常范围 2-8℃）
   - 点击【登记并创建批次】
3. 跳转进入批次详情页 → 点击【发起交接】→ 指定接收人为 **王接收（receiver1）** → 确认
4. **登出**，登录 **receiver1 / receiver123**（接收员）
5. 进入批次详情，点击【签收】→ 为两个样本填写签收温度（如 4.2℃ / 3.8℃）→ 确认
6. ✅ 验证点：
   - 批次状态变为 **全部签收**；样本状态变为 **已签收**；保管人变为「王接收」
   - 每个样本的**交接链**按时间线展示：登记 → 发起交接 → 签收
   - **审计历史**（管理员查看）有 3 条成功记录
   - 签收时的温度记录已入库

### 2️⃣ 场景二：退回重交（含时间线持久化验证）

> ⚠️ **硬约束**：仅 IN_TRANSIT（交接中）状态的样本可以退回，已签收样本**不可退回**。

1. 登录 **sampler1**，新建一批次并发起交接给 **王接收（receiver1）**（样本状态 = IN_TRANSIT）

7. 重新登录 **sampler1**，进入同批次 → ✅ **退回记录和完整时间线依然存在**

## 🔒 安全与合规设计要点

1. **原子事务**：所有操作先做全部校验（角色、状态流、权限、温控），任一条件不满足则整体失败，**不写部分数据**。
2. **状态流转矩阵**：`REGISTERED → IN_TRANSIT → RECEIVED（终态，不可退回）或 RETURNED（可重交）→ VOIDED（终态）`。签收后**硬锁定**，不能退回、不能改交接信息。
3. **失败必审计**：替签、缺原因、缺温度、状态错误都以 `success=false` 写入审计，附失败原因。
4. **无删除语义**：管理员仅能**作废**标记，历史数据永久保留。
5. **温控与交接强绑定**：登记、签收、退回、重新交接**均要求温度记录**；交接前会再次校验所有样本已有温控记录。
6. **重启一致性**：所有数据落盘 `data/db.json`，包含完整交接链数组，重启后按时间戳排序展示。

## 🔧 已修复的硬约束问题

### 修复1：已签收样本不可退回

**问题**：状态流矩阵中 `RECEIVED` 允许转移到 `RETURNED`，`returnBatch()` 也允许 `RECEIVED` 状态的样本退回。导致已签收的样本可以被退回，破坏了签收的终态语义。

**根因**：`STATUS_FLOW[RECEIVED]` 包含 `'RETURNED'`；`returnBatch()` 中 `sample.status !== 'IN_TRANSIT' && sample.status !== 'RECEIVED'` 条件允许 RECEIVED 退回。

**修复**（最小改动 3 处）：
1. `api/services.ts`：`STATUS_FLOW[RECEIVED]` 从 `['RETURNED', 'VOIDED']` 改为 `['VOIDED']`
2. `api/services.ts`：`returnBatch()` 中 `sample.status` 校验从 `!== 'IN_TRANSIT' && !== 'RECEIVED'` 改为 `!== 'IN_TRANSIT'`
3. `src/pages/BatchDetail.tsx` + `QuickAction.tsx`：前端退回按钮仅在 `batch.status === 'IN_TRANSIT'` 时显示；样本选择器中退回操作只允许 `IN_TRANSIT` 状态

**复现脚本**（修复前会成功退回已签收样本，修复后返回 400 错误）：
```bash
# 1. sampler1 登记样本并发起交接
# 2. receiver1 签收（状态 = RECEIVED）
# 3. receiver1 尝试退回 → 修复前：成功；修复后：400 "已签收样本不可退回"
curl -X POST http://localhost:3001/api/labs/batches/{batchId}/return \
  -H "Content-Type: application/json" -b cookie.txt \
  -d '{"sampleIds":["..."], "reason":"测试", "temperatureRecords":[...]}'
```

**验收标准**：已签收退回 → 400 错误；样本状态仍为 RECEIVED；交接链无新节点；审计有失败记录。

### 修复2：CSV 导出连接被关闭

**问题**：`/export/batches` 和 `/export/samples` 路由使用 `res.write('\uFEFF')` + `res.send(csvContent)` 组合，Express 在 `write()` 后调用 `send()` 会重新设置 Content-Length 为仅 csvContent 的长度（不含 BOM），导致响应体与声明长度不匹配，连接被提前关闭，客户端只收到 BOM 3 字节。

**根因**：`res.write()` + `res.send()` 的组合在 Express 4 中行为不一致，Content-Length header 未包含 BOM 的长度。

**修复**（2 处）：
- `api/routes/audit.ts`：两处导出路由都从 `res.write(BOM) + res.send(csvContent)` 改为 `res.end(BOM + csvContent)` 一次性输出

**验收标准**：导出文件 > 100 字节，内容包含批次/样本/温控/交接链四张表，筛选条件在文件头注释中。
